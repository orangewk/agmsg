# リモートトランスポート — 環境をまたぐメッセージング

*[English](remote.md)*

agmsg はローカルファーストであり、メッセージは1台のマシン上の sqlite ストアに存在する。**リモートトランスポート**はそのストアを、プライベートな git リポジトリ(「バス」)を介して環境間 — ラップトップ、デスクトップ、Claude Code on the web、Codex クラウドサンドボックス — でレプリケートし、どこにいるエージェント同士でもメッセージを送り合えるようにする。

設計は [ADR 0005](adr/0005-remote-transport-git-bus.md) に記録されている。git をトランスポートに選んだのは、それがすべてのエージェント環境にすでに存在する唯一の認証済みチャネルだからである。outbound HTTPS しか許されないサンドボックスはデーモンも受信接続も持てないが、プライベートリポジトリの clone/push は全環境でできる。

## セットアップ

1. バスとして使う**プライベート**リポジトリを作る(例: `yourname/agmsg-bus`)。メッセージ本文が履歴に残るため、必ずプライベートにし、GitHub Actions は無効化しておくこと(何も実行される必要がない)。
2. 各環境でそのリポジトリへの `git push`/`pull` が通ることを確認する(バスリポジトリだけにスコープした fine-grained PAT か deploy key。マネージドサンドボックスではセッションのリポジトリスコープにバスリポジトリを追加する)。
3. 各環境のストアをバスに紐付ける:

```bash
~/.agents/skills/agmsg/scripts/remote.sh add git@github.com:yourname/agmsg-bus.git
```

`add` の時点でストアに既に存在するメッセージはローカルに留まる。バスへの接続は「これから先を共有する」ことであり、「過去ログを恒久的な git 履歴に公開する」ことではない。過去分も共有したい場合(マシン移行・バックアップ)は `--include-history` を付ける。

これで完了。以後 `send.sh` は新しいメッセージをバスへ push する。Unix ではバックグラウンドで実行する。Windows/MSYS では、管理されたシェルがコマンド終了時にバックグラウンドの子プロセスを終了させることがあるため、フォアグラウンドで実行する。したがって Windows での `send.sh` 成功は、イベントが bus へ到達したことまでを意味する。他の環境でもフォアグラウンド送信を強制するには `AGMSG_REMOTE_PUSH_SYNC=1`、明示的にバックグラウンド送信へ戻すには `0` を設定する。Stop フックの inbox チェック(および `inbox.sh`)は読み取り前にリモートのメッセージを pull するため、環境をまたぐメッセージもローカルとまったく同じ配信経路で届く。

## コマンド

```bash
remote.sh add <git-url> [--include-history]
                          # このストアをバスリポジトリに紐付ける(clone する)。
                          # 既存メッセージはフラグ指定時のみ共有される
remote.sh status          # url、env id、未エクスポート/未 push 件数
remote.sh sync            # pull + import してから export + push
remote.sh pull            # リモートのイベントをローカルストアに取り込む
remote.sh push            # ローカルのメッセージをエクスポートして push する
remote.sh remove          # バスへの紐付けを解除(ローカルのメッセージは残る)
```

`sync` は手動で往復同期を強制したいときに使う。フックは自動で `pull`/`push` を呼ぶ。

## 使い捨て環境(クラウドサンドボックス)

回収されたコンテナはストアもアイデンティティもバス設定も忘れる。環境のセットアップスクリプト(または SessionStart フック)に冪等な1行を置いておけば、新しいコンテナは毎回同じエージェントとして帰ってくる:

```bash
~/.agents/skills/agmsg/scripts/remote.sh bootstrap git@github.com:you/agmsg-bus.git \
  --team myteam --agent cloudy --env-id cloud-main
```

ストアの初期化、チームへの join、バスへの紐付け、そして死んでいた間に届いたメッセージの pull までを行う。`--env-id`(または `AGMSG_ENV_ID`)で固定した id により、再生成された環境は自分の writer ファイルに追記を再開し、使い捨て id でバスを散らかさない。**固定 id につき同時に生かすインスタンスは1つまで** — 2つ同時に動かすと writer ファイルを共有してしまい、無コンフリクト保証が崩れる。

## 仕組み

- 各環境は `add` 時に **env id**(`<ホスト名>-<ランダム>`)を得て、バス上では自分専用の writer ファイル `events/<team>/<env-id>.<YYYYMM>.jsonl` にのみ追記する。writer ファイルが分離されているため、2つの環境が git の内容コンフリクトを起こすことは構造的にありえない。push は ref の競合しか起きず、自動の rebase + リトライで解決される。
- イベントはグローバルに一意な ID を持ち、インポートは一意インデックスに対する `INSERT OR IGNORE` である。どのファイルを何度リプレイしてもメッセージは重複せず、オフラインだった環境は次回の同期で追い付く。
- ストアには nullable なカラム(`uuid`、`origin`)が2つ追加される。既存の行や、リモートを設定しないストアには影響しない。

## レイテンシとリミット

配信レイテンシは、受信側の pull 周期 + git の往復1回。turn モードでは Stop フックのクールダウン(`delivery.turn.check_interval`、デフォルト 60 秒)、monitor モードではウォッチャーのリモート pull 間隔(`delivery.monitor.remote_pull_interval`、デフォルト 60 秒。ローカル DB の高速ポーリングはローカルのまま)。git の fetch/push トラフィックは GitHub REST API のレートリミットとは別枠で計測され、エージェントチーム規模のメッセージ量なら濫用しきい値まで桁が2〜3個足りないほど余裕がある。詳細な分析は ADR を参照。

## クラウドセッションを起こす (通知層, ADR 0006)

ポーリングは「すでに動いているエージェント」向けの仕組みで、ターン駆動のクラウドセッション(例: Web 版 Claude Code)は誰かがターンを与えない限りバスのメッセージに気づけない。通知層がこのギャップを埋める: バス上の**ルーター**が、新着メッセージの**宛先の購読者だけ**を起こす — 宛先以外は起こさない、既読レシートでは起こさない、イベントを書いた環境自身も起こさない。

すべてのイベント型はクラスを宣言しており、ルーターはこの表に従う:

| クラス | イベント型 | 誰かを起こす |
|---|---|---|
| 配達物 | `message_sent`(および `type` の無い旧形式行) | 宛先のみ |
| 状態ゴシップ | `message_read` | 誰も起こさない(次の sync で収束すればよい) |

セットアップ(GitHub ホストのバスごと):

1. [`docs/examples/agmsg-router.yml`](examples/agmsg-router.yml) を bus ブランチの `.github/workflows/agmsg-router.yml` として commit する。
2. 起こされたいセッションごとに、**そのセッションが watch している open PR** を用意する(差分が自明な常設ドラフト PR でよい。マージ・クローズしないこと)。Claude Code セッションなら PR を「watch して」と頼む。
3. 受信者をバスに登録する:

```
remote.sh subscribe <team> <agent> --pr <number>
remote.sh unsubscribe <team> <agent>
```

これはバスに `subscribers/<team>/<agent>.json` — `{"env": ..., "filter": {"team", "to"}, "wake": {"kind": "pr-comment", "pr": N}}` — を書く。これがルーターのルーティング表のすべて。その team/agent 宛の配達物が(**別の**環境によって)バスに載ると、ルーターは登録された PR にメタデータのみのコメント(件数と team/agent 名。本文は決して載せない)を投稿し、前回の自分のコメントは消すので PR は常に1コメントに保たれる。webhook が watch 中のセッションを起こし、起きたセッションは通常どおり pull して inbox を読む。

登録は (team, agent) につき1件: 複数環境にレプリケートされた同一エージェントは wake 先も1つを共有する(既読状態を共有するのと同じモデル)。`wake.kind` が差し替え点で、ホスト側に wake API ができたら登録エントリを書き換えるだけで移行できる。

セキュリティ上の注意: ルーターはバスの内容を checkout も実行もせず、サードパーティ Action も使わず、出力はメタデータのみ。ただし bus ブランチへの push 権限があれば書き換えられる — これは下記のトラストモデル(push 権限 = バスへのフルアクセス)と整合的。専用バスリポジトリで Actions を無効化していた場合(推奨)、この1本だけ許可するのが wake 対応とのトレードオフになる。公開リポジトリでは購読者名や wake コメントも(バス同様)公開である点に注意。

## トラストモデル: 1バス = 1信頼ドメイン

バス上のものは、そのバスに紐付いた**すべての**環境にレプリケートされる。インポートは全チームの writer ファイルを読み、リポジトリへの push 権限を持つ者は任意の `from` 名でイベントを書ける。つまり**リポジトリのアクセス権こそが認証**である。具体的には:

- あるストアをバスに紐付けてよいのは、「そのバス上の全環境が、そのストアの全チームのメッセージを見てよい」場合だけ
- 2つのエージェント群を互いに隔離したいなら、1つのバスに2チームを載せるのではなく、**バスリポジトリを分ける**(同一マシンで共存させるなら `AGMSG_STORAGE_PATH` でストアも分ける)
- バス上のチームごとのディレクトリは整理と将来のフィルタリングのための名前空間であり、セキュリティ境界ではない

## 既知の制限 (MVP)

- **既読状態はベストエフォートでレプリケートされる**。`inbox.sh` または `check-inbox.sh` が既読にすると、読んだ環境が1件の `message_read` イベントを出力する。他の環境はローカルの `read_at` が未設定のときだけそのレシートを反映するため、単一人格として複数環境に参加しても既読状態は収束する。読んだ側はバックグラウンドで push し、他環境は次回 pull で取り込む。この間は短時間の二重配達が残る。
- チームのメンバーシップ(`teams/` の設定)はレプリケートされない — 各環境で同じチーム名・エージェント名で join すること。
- バスリポジトリの履歴は永続的な記録である。メッセージを本当に削除するには履歴の書き換えが必要になる。
